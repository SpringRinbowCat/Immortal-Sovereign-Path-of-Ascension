# 平台抽象层设计（Platform Abstraction）

> 状态：设计稿 v2（路径抽象四平台实现已就位,与 `src/game` 一致）
> 目标平台：Windows、macOS、iOS、Android。目标：上层代码零平台感知，平台差异全部收敛到本层。

## 1. 目标与原则

- 一套业务代码（Model / Controller / Data / Config）跨各端复用，**业务层不出现 `#ifdef` 平台分支**。
- 平台差异只允许出现在：① 平台实现文件（按平台编译）；② CMake。
- 编译期按目标平台选中对应实现，运行期通过抽象接口使用。

## 2. 需要抽象的平台差异

| 差异 | 说明 | 归属 |
| --- | --- | --- |
| 可写数据目录 | 存档 / 设置文件的落地目录，每平台不同 | 平台路径接口 |
| 资源目录 | 字体 / 图片等 assets 的基准目录，打包方式不同 | 平台路径接口 |
| 文本输入 | 桌面物理键盘 vs 移动端屏幕软键盘 | View 输入适配 |
| 指针输入 | 桌面鼠标 vs 移动端触摸 | SFML 已把单点触摸映射为鼠标事件，命中逻辑可复用 |

## 3. 路径抽象接口（Delegate 层）

```cpp
// IPlatformPaths.h

// 平台相关目录查询接口
#pragma once

#include <filesystem>

class IPlatformPaths {
public:
    virtual ~IPlatformPaths() = default;

    // 只读资源基准目录,字体/图片等 assets 从此解析
    virtual std::filesystem::path resourceDir() const = 0;

    // 每用户可写数据目录,存档/设置落地于此
    virtual std::filesystem::path writableDataDir() const = 0;
};
```

- 接口只依赖标准库 `<filesystem>`，不依赖 SFML。
- 上层（ArchiveService、ResourceManager 的装配方）只持该接口。

## 4. 各平台实现与目录

平台实现合并在**单个文件** `Infra/Platform/PlatformPaths.cpp` 内，用 `#if defined(__ANDROID__) / _WIN32 / __APPLE__`（Apple 再以 `TargetConditionals` 区分 iOS/macOS）分支选择，仅当前平台分支参与编译。

| 平台 | resourceDir | writableDataDir |
| --- | --- | --- |
| Windows | 可执行文件同级 `assets/` | `%APPDATA%\ImmortalSovereign\` |
| macOS | App Bundle 内 `Resources/`，开发期回退到可执行同级 `assets/` | `~/Library/Application Support/ImmortalSovereign/` |
| iOS | App Bundle `Resources/` | App 沙盒 `Documents/` |
| Android | 空路径（APK 内 assets 无文件系统路径，经 SFML/AAssetManager 以相对路径访问） | `ANativeActivity::internalDataPath`（应用内部存储） |

- 单一实现类 `PlatformPathsImpl`（每平台分支各定义一份，只编一份），实现 `IPlatformPaths`（单继承）。
- 工厂 `createPlatformPaths()`（命名空间内非成员函数）在文件末尾统一定义，返回 `std::unique_ptr<IPlatformPaths>`。
- Android 实现依赖 SFML 的 `NativeActivity` 获取内部存储路径（属 Infra，允许依赖 SFML）。
- 目录不存在时由使用方（写入前）创建；查询接口只负责给出路径。
- Android assets 与 iOS 一样无法用绝对桌面路径直读：Android 经 SFML 的 AAssetManager 用相对路径加载，故 `resourceDir()` 返回空，由资源加载方按相对路径交给 SFML 解析。

## 5. 资源策略

- 中文字体 **思源黑体 `SourceHanSansSC-Regular.otf`（SIL OFL 许可）** 已放入仓库 `assets/fonts/`，随可执行/Bundle 分发。
- 加载路径 = `IPlatformPaths::resourceDir()` + `ResourceConfig::kUiFontRelativePath`（`fonts/SourceHanSansSC-Regular.otf`）；由 `ResourceManager::loadFont` 按 key 缓存，View 按 key 取用。
- macOS 的 `resourceDir()`：App Bundle 内取 `Contents/Resources`，开发期（非 Bundle）回退到可执行同级 `assets/`；**无系统字体回退** —— 早期写死 `/System/Library/Fonts/...` 的技术债已清除。

## 6. 输入适配（iOS 重点）

- **指针**：SFML 在移动端把单点触摸映射为鼠标事件，现有"点击命中输入框/按钮"逻辑可直接复用，无需改动。
- **软键盘**：iOS 无物理键盘。需在输入框获得焦点时调 `sf::Keyboard::setVirtualKeyboardVisible(true)` 唤起系统软键盘，失焦时隐藏。文本仍经 `TextEntered` 事件进入，现有过滤逻辑复用。
  - 该调用在桌面端是空操作，可无条件调用，避免平台分支侵入 View 逻辑。
- **回车/Tab**：iOS 软键盘的"完成/换行"键同样产生事件，登录提交逻辑可复用。

## 7. CMake 多平台结构

- 平台路径实现是**单个源文件** `Infra/Platform/PlatformPaths.cpp`，无条件加入 target，平台分支在文件内部 `#if` 选择 —— CMake 不按平台增删该源文件。
- 当前 `CMakeLists.txt` 按 `CMAKE_SYSTEM_NAME` 分两支：
  - `iOS`：打 App Bundle，链接 `SFML::Main`，用 `Info.plist`，把字体打进 Bundle 的 `fonts/`，需 Xcode + 代码签名，且 SFML 为 iOS 构建静态库。
  - `else()`（桌面 macOS / Windows / Linux）：链接系统安装的 SFML，并把 `assets/` 拷到可执行同级。
- **Android**：SFML 的 Android 库经独立脚本交叉编译，但游戏 App 的 Android 打包目标**尚未接入主 `CMakeLists.txt`**（见验证边界第 9 节）。

## 8. 分层归属与装配

- 接口 `IPlatformPaths` → `Delegate/`。
- 平台实现 + 工厂 → `Infra/Platform/`（Director 持有的基础设施，纯标准库、不依赖 SFML）。
- Director 装配期：
  1. `createPlatformPaths()` 得到平台路径对象。
  2. 把 `resourceDir()` 交给 `ResourceManager`（解析字体/图片）。
  3. 把 `writableDataDir()` 交给 `ArchiveService`（存档读写，后续模块）。

## 11. 屏幕适配（虚拟分辨率 + Letterbox）

为适配桌面任意窗口尺寸与移动端各种分辨率/方向，UI 统一在**固定虚拟分辨率**（当前竖屏 720×1280，由 `WindowConfig` 提供）下设计坐标，运行期等比缩放到实际屏幕：

- Director 持有一个 `sf::View`（大小=虚拟分辨率），通过 `setViewport` 把它等比缩放铺到窗口，多余区域居中留黑边（letterbox），不拉伸变形。
- 窗口 `Resized` 事件触发重算视口；移动端全屏尺寸同理适配。
- 输入坐标经 `window.mapPixelToCoords` 从物理像素反算回虚拟坐标，缩放后点击/触摸命中仍准确。
- 该机制对桌面（可调窗口）与移动端（固定全屏、不同宽高比）通用，View 内部只认虚拟坐标，无需感知真实分辨率。
- 实现位置：`Director`（持有 `sf::View`、处理 `Resized`、`_applyLetterbox`）；各场景 View 的 `handleEvent` 接收窗口引用用于坐标换算。
